# Plan — M1.3 Snapshot ordering & trigger scope

## Goal

Codify the PRD contract: **snapshots are taken only on `Allow`, before execution, never for `Block`.**

Done when:
- `shell_flow.rs` and `watch/runner.rs` create snapshots **after** the user/system has approved the command and **before** the (optionally sandboxed) child runs.
- A `Block` outcome produces no snapshot records and writes an audit entry with an empty `snapshots` array.
- An integration test proves that a sandboxed `Danger` command has a snapshot recorded before it runs, and that a `Block`-level command records no snapshots.
- `docs/config-schema.md` and any other user-facing docs explicitly state the snapshot-before-exec / no-snapshot-on-block contract.

## Current state

In `shell_flow.rs::run_planned_shell_command`:
- `ExecutionDisposition::Execute`: snapshot created, audit appended, then `exec_command`.
- `ExecutionDisposition::RequiresApproval`: snapshot created **before** `show_confirmation_decision`, audit appended after the prompt, then `exec_command` if approved.
- `ExecutionDisposition::Block`: no snapshot, audit appended with `&[]`.

This means in the prompt path the snapshot is taken **before** approval, violating the PRD. The
`Block` path already has no snapshots.

In `watch/runner.rs::run_watch_plan`:
- `create_watch_snapshots` is called once at the top of the function, before the disposition switch.
- The snapshots are then used for the audit entry regardless of whether the decision ends up `Blocked`.

This means `Block` commands in watch mode can end up with a snapshot being created, and snapshots are taken before approval.

`policy_output.rs` JSON mode is evaluation-only and never executes; it does not create snapshots and must continue not to.

## Key design decisions

1. **Move snapshot creation to the allowed-execution branch.**
   - In `shell_flow.rs` only call `create_snapshots_for_plan` once the final decision is `Approved`/`AutoApproved` and before `exec_command`.
   - In `watch/runner.rs` call `create_watch_snapshots` only inside the `Approved | AutoApproved` branch before `execute_and_emit`.

2. **Audit entry must be written after the snapshot is created but before the child runs.**
   - The existing audit append already happens after snapshot creation; we just need to ensure the snapshot variable is populated in the allowed branch.

3. **Deny / Block paths keep empty `snapshots`.**
   - Denied prompts and hard blocks must not create snapshots.

4. **The prompt needs to show *that* a snapshot will be taken, not necessarily the recorded id.**
   - `show_confirmation_decision` currently receives `&snapshots` for display. After the change the snapshots do not yet exist at prompt time.
   - The TUI already renders the snapshot plan (applicable plugins) from the explanation. The snapshot records are only used as a list; showing an empty list in the prompt is acceptable because the explanation still lists `applicable_snapshot_plugins`. Alternatively, we can pass a preview list built from `plan.snapshot_plan().applicable_plugins()` — but we must avoid doing real I/O before approval.
   - Decision: keep the prompt display simple by passing `&[]` for snapshots. The explanation already carries `applicable_snapshot_plugins`.

5. **Watch mode snapshots move from pre-decision to post-approval.**
   - `create_watch_snapshots` currently runs before the decision. Move it into the approved branch.
   - The audit append in the non-approved branches must use an empty snapshot list.

## Files to change

### `aegis` root binary

- `src/shell_flow.rs`
  - Refactor `run_planned_shell_command`:
    - Do not create snapshots in the `RequiresApproval` branch before the prompt.
    - After the prompt, if approved, create snapshots, append audit with `Decision::Approved`, then `exec_command`.
    - If denied, append audit with `Decision::Denied` and `&[]` snapshots.
    - `Execute` branch remains snapshot → audit → exec.
    - `Block` branch remains no snapshot.
  - Keep the existing `#[cfg(test)]` helpers (`decide_command`, `execute_policy_decision`, `test_command_explanation`, `evaluate_policy_decision`) compatible with the new ordering. These helpers currently create snapshots before the prompt for test convenience. We can either:
    a) keep the test helper behavior unchanged (tests simulate the old order), or
    b) update the helper to match production ordering.
    - Decision: update the test helper to match production ordering so tests do not lie about the contract. The helper functions are internal to tests and the change is mechanical.

- `src/watch/runner.rs`
  - Remove the top-level `create_watch_snapshots` call.
  - Inside `run_watch_plan`, when `runtime_decision` is `Approved | AutoApproved`, call `create_watch_snapshots` before `execute_and_emit`, then append audit with the snapshots.
  - For `Denied`/`Blocked`/other branches, append audit with an empty snapshot list.

- `src/runtime/context.rs` (maybe)
  - No changes needed; `create_snapshots` and `create_snapshots_async` already do the right thing when called.

- `docs/config-schema.md`
  - Update the "Snapshot policy" section to add:
    - Snapshots are created only after a `Danger` command is approved.
    - Blocked commands never create snapshots.
    - Snapshots happen before the command is executed (and before sandbox confinement is applied, when configured).

## Detailed implementation steps

### Step 1 — Update `src/shell_flow.rs`

1. In `run_planned_shell_command`, change the `RequiresApproval` branch:
   - Remove the early `create_snapshots_for_plan` call.
   - Run the prompt.
   - If approved, create snapshots, append audit with `Decision::Approved` and snapshots, then `exec_command`.
   - If denied, append audit with `Decision::Denied` and `&[]` snapshots, return `EXIT_DENIED`.

2. Keep `Execute` branch as snapshot → audit → exec.
3. Keep `Block` branch with `&[]` snapshots.

4. Update the `#[cfg(test)]` helper `execute_policy_decision` to match:
   - For `AutoApprove`: create snapshots and return them.
   - For `Prompt`: prompt first, then create snapshots only if approved.
   - For `Block`: return empty snapshots.

### Step 2 — Update `src/watch/runner.rs`

1. Remove the top-level `let snapshots = create_watch_snapshots(...)`.
2. After computing `runtime_decision`, append audit:
   - If approved, first create snapshots, then append audit with snapshots, then `execute_and_emit`.
   - Otherwise, append audit with empty snapshots, then emit the result frame.
3. Add helper to create snapshots only for approved decisions.

### Step 3 — Update docs

1. `docs/config-schema.md` snapshot policy section:
   - Add bullets about ordering and trigger scope.
2. Optionally update `docs/adr/adr-004-snapshots-are-best-effort-audit-is-append-only.md` to mention ordering.

### Step 4 — Tests

1. Add unit tests in `src/shell_flow.rs` under `#[cfg(test)]`.
2. Add `tests/snapshot_ordering.rs` integration tests.
3. Run full test suite.

## Open questions / risks

1. **Prompt display without snapshots.** The prompt currently shows snapshot records. Passing `&[]` means the operator will not see snapshot ids before approval. The explanation still lists applicable plugins. Is this acceptable per UX? Yes — the PRD only requires the snapshot to happen before exec; pre-approval snapshot ids were a side effect, not a requirement.

2. **Watch-mode audit append ordering.** Moving snapshot creation after decision means the audit append for approved commands happens after snapshot creation (good). The result frame is emitted after the child exits. We need to ensure the audit entry is appended before the result frame to preserve the "snapshot existed before execution" proof in tests.

3. **Test environment sandbox availability.** Proving "before sandboxed execution" requires a sandbox. If the test runner is Linux/macOS with bwrap/sandbox-exec available, we can enable sandbox. Otherwise, we prove the snapshot record timestamp precedes the child output/exit, which is sufficient for the ordering contract.

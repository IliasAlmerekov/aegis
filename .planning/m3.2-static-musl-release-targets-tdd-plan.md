# M3.2 TDD Plan — Static musl release targets

## Task

Implement `TASKS.md` milestone **M3.2 — Static musl release targets**.

The release workflow must publish statically portable Linux binaries for:

- `x86_64-unknown-linux-musl`
- `aarch64-unknown-linux-musl`

and must stop publishing Linux GNU-targeted release binaries:

- `x86_64-unknown-linux-gnu`
- `aarch64-unknown-linux-gnu`

This task touches CI/release workflow files, so the actual implementation requires
explicit human sign-off before modifying `.github/workflows/release.yml`.

## Constraints

- All shell commands must be run through `rtk`.
- Do not change `Cargo.toml`, `Cargo.lock`, `deny.toml`, or CI workflows without
  explicit human approval.
- Keep installer asset names stable unless there is a separate approved migration:
  - `aegis-linux-x86_64`
  - `aegis-linux-aarch64`
- Do not add dependencies.
- Do not touch parser/scanner hot paths.
- Do not weaken release checksum generation.
- Prefer tests that verify the release contract without relying on network access.

## Definition of Done

M3.2 is complete when:

1. `.github/workflows/release.yml` builds Linux release artifacts with:
   - `x86_64-unknown-linux-musl`
   - `aarch64-unknown-linux-musl`
2. `.github/workflows/release.yml` no longer contains Linux GNU release targets:
   - `x86_64-unknown-linux-gnu`
   - `aarch64-unknown-linux-gnu`
3. `aarch64-unknown-linux-musl` is built via `cross`.
4. Linux release artifacts are verified as static before checksum generation and
   artifact upload.
5. `.sha256` sidecars are still generated for every release artifact.
6. Installer-facing asset names remain unchanged.
7. Regression tests fail if GNU targets are reintroduced or musl targets are
   removed.
8. Relevant local gates pass:
   - `rtk cargo fmt --check`
   - `rtk cargo clippy -- -D warnings`
   - `rtk cargo test`

## Iteration 2 — GREEN: switch release matrix to musl

### Goal

Make the smallest workflow change that satisfies the target matrix tests.

### Change

In `.github/workflows/release.yml`, replace Linux target entries.

Before:

```yaml
- target: x86_64-unknown-linux-gnu
  os: ubuntu-latest
  asset_name: aegis-linux-x86_64

- target: aarch64-unknown-linux-gnu
  os: ubuntu-latest
  asset_name: aegis-linux-aarch64
  use_cross: true
```

After:

```yaml
- target: x86_64-unknown-linux-musl
  os: ubuntu-latest
  asset_name: aegis-linux-x86_64
  use_cross: true

- target: aarch64-unknown-linux-musl
  os: ubuntu-latest
  asset_name: aegis-linux-aarch64
  use_cross: true
```

### Rationale

- `aarch64-unknown-linux-musl` should use `cross`.
- `x86_64-unknown-linux-musl` should also use `cross` to avoid depending on
  runner-specific musl linker setup.
- Keeping both Linux targets on `cross` makes the release matrix more uniform.
- Asset names stay stable, so the installer contract remains intact.

### GREEN command

```bash
rtk cargo test --test release_workflow
```

## Iteration 4 — GREEN: add static verification step

### Goal

Fail the release job if a Linux musl artifact is dynamically linked.

### Placement

Add the step after `Rename binary` and before `Generate SHA256 checksum`.

### Workflow step

```yaml
- name: Verify static Linux binary
  if: contains(matrix.target, 'unknown-linux-musl')
  shell: bash
  run: |
    file "${{ matrix.asset_name }}"
    if ldd "${{ matrix.asset_name }}" 2>&1 | grep -q "not a dynamic executable"; then
      exit 0
    fi
    echo "expected static Linux binary, but ldd found dynamic dependencies" >&2
    ldd "${{ matrix.asset_name }}" >&2
    exit 1
```

### Rationale

- Runs only for Linux musl targets.
- Fails before checksum generation and upload.
- Uses standard Linux tooling available on GitHub-hosted Ubuntu runners.
- Does not add dependencies.

### GREEN command

```bash
rtk cargo test --test release_workflow
```

## Iteration 5 — Local build smoke checks

### Goal

Catch obvious target/build regressions locally where possible.

### Preferred checks

If the local environment has the target/linker support:

```bash
rtk rustup target add x86_64-unknown-linux-musl
rtk cargo build --release --target x86_64-unknown-linux-musl
```

If `cross` is available locally:

```bash
rtk cross build --release --target x86_64-unknown-linux-musl
rtk cross build --release --target aarch64-unknown-linux-musl
```

### Handling local environment failures

If local builds fail because `cross`, Docker, or musl linker tooling is missing,
do not work around this by adding repository dependencies. Record it as a local
environment limitation and rely on the release workflow job for the authoritative
cross-build verification.

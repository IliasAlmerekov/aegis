# Ticket 1.3: SnapshotRegistry config-aware - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Make `SnapshotRegistry` honour `auto_snapshot_git` / `auto_snapshot_docker` config flags ‚Äî so that when a flag is `false`, the corresponding plugin is never registered and no snapshot is created. Audit and UI must only reflect snapshots that actually ran.

</domain>

<decisions>
## Implementation Decisions

### Default implementation

- **D-01:** Keep `impl Default for SnapshotRegistry` as-is. It delegates to `from_config(&Config::default())` which is already safe. No callers in production paths use it ‚Äî `RuntimeContext::new` always calls `from_config` explicitly.

### Core wiring

- **D-02:** `from_config` and `RuntimeContext` are already correct ‚Äî no changes to `src/snapshot/mod.rs` logic or `src/runtime.rs`.
- **D-03:** UI (`src/ui/confirm.rs`) and audit (`src/audit/logger.rs`) already correctly reflect only the snapshots that were actually created ‚Äî no changes needed.

### Missing gap ‚Äî integration tests

- **D-04:** Add integration tests in `tests/full_pipeline.rs` covering:
  - `auto_snapshot_git = false` in `.aegis.toml` ‚Üí git stash is NOT called (no snapshot record in audit)
  - `auto_snapshot_docker = false` in `.aegis.toml` ‚Üí docker commit is NOT called (no snapshot record in audit)
- **D-05:** Tests must run against a real built binary (same pattern as other tests in `full_pipeline.rs`), not mocks.
- **D-06:** The git snapshot test must set `current_dir` to a temp dir that is NOT a git repo, OR a temp git repo ‚Äî so that `GitPlugin::is_applicable` returns the expected value. Use a non-git temp dir with `auto_snapshot_git=false` to confirm the plugin was never registered (not just skipped by `is_applicable`).

### Claude's Discretion

- Exact test helper structure (fixture setup, assertion style) ‚Äî follow the patterns already established in `full_pipeline.rs`.

</decisions>

<code_context>

## Existing Code Insights

### Reusable Assets

- `base_command(home)` helper in `full_pipeline.rs` ‚Äî sets up `AEGIS_REAL_SHELL`, `AEGIS_CI=0`, `HOME` env. Use this for new tests.
- `read_audit_entries(home)` ‚Äî parses `~/.aegis/audit.jsonl` into `Vec<Value>`. Check `entry["snapshots"]` to verify no snapshots were recorded.
- `write_executable(path, body)` helper ‚Äî for stubbing out `git` or `docker` binaries in PATH if needed.

### Established Patterns

- Tests write `.aegis.toml` into a `TempDir` workspace and pass `current_dir(workspace.path())` to the command. Same pattern needed here.
- `audit.jsonl` entries have a `"snapshots"` array ‚Äî when no snapshots were created it will be empty (`[]`).
- The `AEGIS_FORCE_INTERACTIVE=1` env var forces the interactive dialog even without a TTY ‚Äî use it for Danger commands in tests.

### Integration Points

- `SnapshotRegistry::from_config` is called inside `RuntimeContext::new` ‚Üí `RuntimeContext::load` ‚Üí called from `main.rs`. So config flags set in `.aegis.toml` flow through to plugin registration automatically.
- Docker snapshot test: `auto_snapshot_docker` defaults to `false`, so a baseline test with no config already covers docker-off. Consider a test that sets it to `true` and stubs `docker` to verify the positive case as well (optional, but strengthens coverage).

</code_context>

<specifics>
## Specific Ideas

- For the git snapshot test: set `auto_snapshot_git = false` in `.aegis.toml`, run a Danger command, pipe "yes" to approve it, then check that `audit.jsonl` entry has `"snapshots": []`.
- For the docker snapshot test: similar pattern, `auto_snapshot_docker = false` (already the default, but make it explicit in the config).
- Consider also testing the positive case: `auto_snapshot_git = true`, git repo present, Danger command approved ‚Üí `"snapshots"` contains a git entry. This proves the end-to-end pipeline, not just the negative path.

</specifics>

# Ticket 1.4 — Full `Mode` Runtime Semantics Design

## Problem

`Mode` (`Protect`, `Audit`, `Strict`) is parsed from config but does not currently affect runtime decisions. This makes configuration partially decorative and behavior unpredictable.

## Scope

Implement real mode-driven policy semantics in the decision layer, with explicit and testable behavior across:

- interactive flow
- CI behavior
- allowlist handling
- snapshot execution
- audit logging
- config UX (`config init` and `config show`)

## Non-goals

- No changes to scanner pattern matching semantics.
- No change to `RiskLevel` ordering or meaning.
- No broad refactor of UI rendering (`ui/confirm.rs`) beyond integration points.

## Confirmed Product Decisions

### 1. Mode precedence

`mode` has higher priority than `ci_policy`.

- `Audit` must not block in CI.
- `Strict` stays strict in CI.
- `ci_policy` remains effective only in `Protect`.

### 2. Strict allowlist override

In `Strict`, allowlist is disabled by default.

Introduce an explicit config flag:

`strict_allowlist_override = false` (default)

When enabled, it may auto-approve only `Warn` and `Danger`; it must never bypass `Block`.

### 3. Snapshots

- `Protect`: snapshots on `Danger` only.
- `Audit`: no snapshots.
- `Strict`: deny non-`Safe`; snapshots are therefore not created in the deny path.

### 4. Block invariant

In blocking modes (`Protect`, `Strict`), `Block` must remain non-bypassable in all CI states and all allowlist states.

In `Audit`, policy is intentionally non-blocking for all risks; allowlist still must not change the mode outcome.

## Architecture

## New decision layer module

Add a dedicated policy module (e.g. `src/decision.rs`) responsible for evaluating runtime policy from pure inputs.

`main.rs` remains orchestration-only:

1. load runtime context
2. assess command
3. call decision evaluator
4. execute resulting plan (prompt, snapshots, command execution, audit append)

This removes policy branching from CLI glue and centralizes mode semantics.

## Decision evaluator contract

Inputs (conceptual):

- `mode`
- `risk`
- `in_ci`
- `ci_policy`
- `allowlist_match` present/absent
- `strict_allowlist_override`

Output (conceptual `DecisionPlan`):

- policy action (`AutoApprove`, `Block`, `Prompt`)
- whether prompt is required
- whether snapshots should run
- whether allowlist was policy-effective

The function should be deterministic and side-effect free so matrix tests can cover all branches.

Two-phase mapping:

1. decision layer returns deterministic policy action (`AutoApprove`, `Block`, `Prompt`)
2. CLI wiring resolves runtime outcome:
   - `AutoApprove` -> `Decision::AutoApproved`
   - `Block` -> `Decision::Blocked`
   - `Prompt` + user yes -> `Decision::Approved`
   - `Prompt` + user no -> `Decision::Denied`

## Policy Matrix

## Protect (baseline)

- `Safe` → auto-approve.
- `Warn` / `Danger` → prompt unless allowlisted.
- `Block` → hard block.
- allowlist can auto-approve only `Warn` / `Danger`.
- snapshots only for `Danger` when execution flow requires it.
- in CI + `CiPolicy::Block`: block non-safe unless allowlisted (`Warn`/`Danger` only), preserving `Block` non-bypass.

## Audit

- Never prompt.
- Never block.
- Always produce an audit record.
- No snapshots.
- allowlist does not alter decision outcome (it may still be attached as context in audit metadata if matched).
- CI does not introduce blocking in this mode.

## Strict

- `Safe` → auto-approve.
- `Warn` / `Danger` / `Block` → blocked by default.
- allowlist disabled by default.
- if `strict_allowlist_override = true`, allowlist can auto-approve `Warn`/`Danger` only.
- `Block` remains blocked regardless of override.
- if `strict_allowlist_override = true` and allowlisted `Danger` is auto-approved, snapshots are created (same danger snapshot rule as `Protect`).
- CI cannot weaken strict behavior.

## Explicit mode x risk behavior table

| Mode | Risk | Policy action | Prompt | Snapshot | Allowlist effect |
|------|------|------------------|--------|----------|------------------|
| Protect | Safe | AutoApprove | No | No | None |
| Protect | Warn | Prompt | Yes | No | AutoApprove when matched |
| Protect | Danger | Prompt | Yes | Yes | AutoApprove when matched; snapshot still created |
| Protect | Block | Block | No | No | No bypass |
| Audit | Safe | AutoApprove | No | No | No decision change |
| Audit | Warn | AutoApprove | No | No | No decision change |
| Audit | Danger | AutoApprove | No | No | No decision change |
| Audit | Block | AutoApprove | No | No | No decision change |
| Strict | Safe | AutoApprove | No | No | None |
| Strict | Warn | Block | No | No | If `strict_allowlist_override=true` and matched: AutoApprove |
| Strict | Danger | Block | No | No | If `strict_allowlist_override=true` and matched: AutoApprove + snapshot |
| Strict | Block | Block | No | No | No bypass |

## Configuration changes

## Config model

Add:

- `strict_allowlist_override: bool` with default `false`

Merge semantics must remain backward-compatible:

- missing field in old config files resolves to default `false`
- layered merge follows existing precedence rules

## `config init`

Update init template comments so behavior is truthful and no longer states `Audit`/`Strict` as not implemented.

Document:

- mode semantics summary
- strict allowlist override behavior and explicit safety warning

## `config show`

Ensure emitted config includes effective value for `strict_allowlist_override` and aligns with runtime-supported behavior.

## Documentation updates

Update README mode reference and quick table to match runtime reality.

Required clarifications:

- `Audit` is dry-run friendly and non-blocking
- `Strict` deny-by-default behavior
- explicit statement that `Block` is never bypassed
- CI interaction governed by mode precedence

## Testing Strategy

## Unit tests (decision layer)

Add matrix coverage for:

- all three modes
- all risk levels
- CI on/off
- `ci_policy` permutations (where applicable)
- allowlist matched/unmatched
- strict override true/false

Must include regression assertions for invariants:

- `Block` is non-bypassable
- `Audit` never blocks
- `Strict` blocks non-safe by default
- `mode` actually changes decision output

## Integration touchpoints

Adjust existing main-flow tests to validate that:

- main wiring consumes `DecisionPlan` correctly
- snapshots run only when requested by plan
- audit entries reflect mode-driven decisions

## Config UX tests

Add/adjust tests asserting:

- init template text reflects implemented behavior
- config serialization/deserialization includes new strict override field with correct default and merge behavior

## Acceptance Criteria Mapping

1. `mode` changes `Decision` at runtime.
2. Tests exist for `Protect`, `Audit`, `Strict`.
3. `config init` and `config show` reflect supported behavior.

## Risks and Mitigations

- **Risk:** policy regressions in CI and allowlist interplay.  
  **Mitigation:** explicit matrix tests and invariants for non-bypassable `Block`.

- **Risk:** drift between docs and behavior.  
  **Mitigation:** update init template + README in same change set.

- **Risk:** policy logic creeping back into `main.rs`.  
  **Mitigation:** decision module as single runtime policy source of truth.

## Implementation notes for the next phase

- Keep `src/main.rs` thin; avoid reintroducing policy branching there.
- Keep scanner hot path untouched and synchronous.
- Preserve append-only audit contract and existing exit-code semantics.

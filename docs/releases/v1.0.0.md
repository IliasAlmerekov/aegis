# Planned Aegis v1.0.0 release summary

This document summarizes the future release contract for the `v1.0.0` target
using the current repository state, workflow definitions, and supporting docs.

The repository's current crate version is `0.5.1`; this document is
forward-looking and should not be read as evidence that `v1.0.0` is already the
active release line.

## Summary

Aegis for the future `v1.0.0` tag is documented as a shell-proxy guardrail
that:

- intercepts shell commands before execution;
- classifies commands into `Safe`, `Warn`, `Danger`, or `Block`;
- requires confirmation for risky commands;
- hard-blocks `Block` commands;
- writes audit data and supports best-effort snapshots for dangerous actions
  when configured.

This release summary does **not** claim stronger guarantees than the current
docs and implementation support.

## Checksum verification path

The release docs now treat the manual checksum-first flow in
[docs/release-readiness.md](../release-readiness.md) as the clearest
verification story:

- download the binary and its matching `.sha256` file from the GitHub Release;
- resolve the expected checksum from the sidecar;
- verify the binary with `sha256sum` when available, or `shasum -a 256` as a
  fallback;
- stop before installation if the checksum is missing or does not match.

The install script remains a convenience path that uses the same checksum sidecar
before writing into `BINDIR`, but this document should not frame it as the
canonical verification story.

## Install and upgrade pointers

### Fresh install

Use the published install script if you want the convenience path:

```sh
curl -fsSL https://raw.githubusercontent.com/IliasAlmerekov/aegis/main/scripts/install.sh | sh
```

The script URL points at `main`; when `v1.0.0` release assets exist, the
release version is pinned separately by setting `AEGIS_VERSION=v1.0.0` when you
want the installer to fetch the `v1.0.0` binary release assets.

The convenience installer is documented as a global-first path:

- it performs the managed shell-wrapper setup rather than offering Local / Binary installer modes
- it rejects the removed `AEGIS_SETUP_MODE` / `AEGIS_SKIP_SHELL_SETUP` controls
- it auto-attempts Claude Code / Codex hook setup only when run from a real local checkout with supported agent directories present
- automatic shell setup currently recognizes `bash` and `zsh`; other shells use the documented `AEGIS_SHELL_RC` / `AEGIS_REAL_SHELL` override path

### Pin to a specific release

To pin the binary version to `v1.0.0`, set `AEGIS_VERSION=v1.0.0` before
running the installer.

### Upgrade

Re-run the installer for the target release version. The current installer is
documented and tested as an idempotent managed-block updater for shell setup.

### Removal

If you need to remove the shell wrapper and binary, use:

```sh
curl -fsSL https://raw.githubusercontent.com/IliasAlmerekov/aegis/main/scripts/uninstall.sh | sh
```

## Known limitations

The current docs describe Aegis as a heuristic guardrail, not a sandbox. Known
limitations remain in scope:

- obfuscated or encoded commands may evade detection;
- runtime-generated commands such as `eval "$(…)"` are out of scope for the
  current interception boundary;
- indirect execution paths, such as writing a script and running it later, are
  not fully covered by the guardrail;
- native Windows shells are unsupported; Windows host usage is only documented
  via WSL2 terminal guidance;
- snapshot and rollback behavior is best-effort and should not be treated as a
  perfect recovery guarantee.

See [docs/threat-model.md](../threat-model.md) and
[README.md](../../README.md) for the current limitation language.

# Pack 05: Compatibility Contracts and Threat-Model Documentation (G5, G6)

**Status:** Passed  
**Start date:** 2026-04-15  
**Related gates:** `G5`, `G6`  
**Blocking:** Pack 05 закрывает `G5` + `G6` (`blocking_v1 = true` для обоих).

## Purpose

Pack 05 делает явным и проверяемым контракт зрелости: поведенческие и эксплуатационные обещания, которые должны удерживаться между v0.x и v1.0, документированы и связаны с тестами. В фокусе:

- конфигурационный контракт (версии, merge/precedence, режимы, allowlist, snapshot policy);
- публичный JSON output + exit-code contract для `aegis`;
- актуальная threat модель и прозрачные ограничения/негативные возможности.

## Scope

### Included

1. **G5:** Config schema + exit/audit-contract docs.
   - Проверка наличия и актуальности `docs/config-schema.md`:
     - `config_version` и схема эволюции;
     - merge order + precedence;
     - режимы (`Protect`/`Audit`/`Strict`) и политика CI;
     - allowlist/allowlist scope semantics;
     - snapshot policy + migration note + `compatibility policy`;
     - JSON output contract (`schema_version`, `risk`, `decision`, `exit_code`, `decision_source`);
     - exit-code compatibility contract (в т.ч. соответствие JSON `exit_code`).
   - Проверка связей:
     - README-ссылка на контракт конфигурации;
     - тесты-доктулпрайс для фиксации наличия полей и wording.

2. **G6:** Threat model as an explicit honesty document.
   - Обновление и верификация `docs/threat-model.md`:
     - инварианты защиты, угрозы, residual risk;
     - non-goals и limitations как отдельный блок;
     - clear statement that Aegis is heuristic and not complete;
     - maturity note (fuzzing coverage state, current gaps).
   - Проверка, что `README.md` продолжает ссылаться на Threat model.

### Excluded

- Любые изменения parser/scanner/snapshot runtime без отдельного gate-pack.
- Добавление новых security guarantees в коде без отдельного gate.
- Расширение threat model с фактическими требованиями к новым механизмам (это делается через Track C, не Track A).

## Acceptance Criteria

### AC1 (G5): config-schema contract is complete and stable

1. `docs/config-schema.md` содержит все перечисленные блоки:
   - schema evolution + `config_version`;
   - merge order + precedence + allowlist ordering semantics;
   - mode semantics + CI policy behavior;
   - snapshot policy + provider behavior/requirements (including current built-in set);
   - JSON contract (top-level fields + required semantics);
   - exit-code compatibility contract with explicit mapping.
2. Для каждой публикуемой единицы интерфейса есть проверка в тестах:
   - `tests/config_schema_docs.rs::config_schema_doc_exists_and_describes_versioning_and_migration`;
   - `tests/config_schema_docs.rs::readme_links_to_config_schema_policy`;
   - `tests/contracts_docs.rs::config_schema_contract_covers_exit_code_compatibility`.
3. `README.md` содержит link на `docs/config-schema.md`.

### AC2 (G5): JSON and runtime exit contracts are mutually consistent

1. `docs/config-schema.md` явно описывает, что JSON поле `exit_code` совпадает с process-level exit для соответствующего сценария.
2. Маппинг exit code (`0`, `2`, `3`, `4`, passthrough) заклеивается в документацию и покрывается regression-тестом документации.
3. Нет противоречивых обещаний в README относительно поведения в отношении blocked/prompt/internal failures.

### AC3 (G6): threat-model is explicit and current

1. `docs/threat-model.md` включает:
   - раздел “Aegis is a heuristic...”, список угроз,
   - `explicit non-goals`,
   - residual risks и known examples для evasions,
   - section trust boundaries/untrusted inputs/reliability notes.
2. Указанные ограничения не конфликтуют с `docs/architecture-decisions.md` и `README.md`.
3. Проверка тестом:
   - `tests/contracts_docs.rs::threat_model_is_current_and_documents_non_goals_honestly`;
   - `tests/contracts_docs.rs::readme_links_to_contract_docs`.

### AC4 (G6): contract freshness and ownership

1. `docs/v1/gate-register.md` для `G5`, `G6` указывает:
   - `status=passed`;
   - `last_verified_at` на актуальную дату после текущего прохода;
   - `owner` и `next_owner` заполнены.
2. `docs/v1/gate-register.md` и mini-spec остаются согласованными по scope и зависимостям.

## Implementation notes

1. Не изменяем контракты runtime в этом pack-е; только документируем и синхронизируем evidence.
2. Для устойчивости к дрейфу: любые правки runtime-интерфейса (audit JSON/exit codes/конфигы) должны обновлять этот pack и тесты документации.

## Rollback / stop conditions

- Если после проверки обнаруживается расхождение docs ↔ code (`exit_code` mapping, schema fields, non-goals), pack переводится в `blocked` до фикса.
- Если изменяются угрозы/доказательная модель (например, новые parser invariants), требуется отдельный Track C gate-обсуждение перед обновлением scope этого pack-а.

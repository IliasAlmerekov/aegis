# Pack 03: Performance Baseline + CI/Security Gates

**Status:** Passed (G3 and G10 completed; G10 unblocked after G8/G9 closure)
**Start date:** 2026-04-15
**Related gates:** `G3`, `G10`
**Blocking:** Pack 03 закрывает `G3` (`blocking_v1 = true`) и завершает `G10` после закрытия зависимостей `G8`/`G9` (`blocking_v1 = true`).

## Purpose

Pack 03 делает воспроизводимым и проверяемым контракт по производительности и
релизной надёжности CI:

- безопасная и повторяемая проверка hot-path latency через `scanner_bench` baseline;
- закреплённый контракт CI/секьюрити-проверок и зависимостей;
- документирование стабильного состояния текущего CI как артефакта для gates `G3` и `G10`.

## Scope

### Включено

1. Проверка и закрепление критических метрик hot-path:
   - `1000_safe_commands`
   - `100_dangerous_commands`
   - `heredoc_worst_case`
2. Проверка связанного CI-пайплайна:
   - pinned Rust/toolchain и pinned external actions/dependencies;
   - наличие и успешность `quality` + `security` + `performance` jobs;
   - наличие release-сборки и checksum-артефактов в release workflow.
3. Формирование и обновление доказательств для gate-регистра:
   - явные команды проверки в `docs/v1/gate-register.md` (evidence);
   - обновление `last_verified_at`/`evidence` после каждого прохода.

### Исключено

- Изменение функциональности Aegis-интерцепции.
- Поднятие новых perf-метрик, которые не относятся к parser/scanner hot-path.
- Работа с artifact signing/attestations (будет в G8).

## Acceptance criteria

### AC1 (G3): baseline benchmark contract is executable and stable

1. `docs/performance-baseline.md` содержит policy/допуск по `scanner_bench` и
   понятную диагностику `aegis_benchcheck`.
2. Наличие `perf/scanner_bench_baseline.toml` с порогами и именами benchmark-ов.
3. Выполнение:
   - `rtk cargo bench --bench scanner_bench`
   - `rtk cargo run --quiet --bin aegis_benchcheck -- --baseline perf/scanner_bench_baseline.toml --criterion-root target/criterion`
   даёт `PASS` по всем tracked benchmark-ам.

### AC2 (G3): budget semantics are documented and reviewable

1. В доке явно закреплён "не должен регрессировать больше чем X%" контракт для каждого
   benchmark.
2. Доказаны результаты последнего прогона с датой проверки и командами.
3. TTL для G3 в register-е остаётся применённым (14 дней).

### AC3 (G10): CI/security gates are documented as locked contracts

1. `.github/workflows/ci.yml` содержит:
   - форматированные проверки: `cargo fmt --check`, `cargo clippy`, `cargo test`;
   - security lane: `cargo audit`, `cargo deny check`;
   - perf lane: `cargo bench` + `aegis_benchcheck` + артефакт `benchmark-report.txt`;
   - release lane: build jobs и release workflow с checksum-artifacts.
2. `.github/workflows/release.yml` содержит determinистично описанный build matrix и
   generation of SHA256 sidecars.
3. `docs/ci.md` и `docs/performance-baseline.md` синхронизированы с текущими workflow.

### AC4 (G10): local validation run as proof before claiming ready

1. Локально собран локальный proof bundle:
   - `rtk cargo fmt --check`
   - `rtk cargo clippy -- -D warnings`
   - `rtk cargo test`
   - `rtk cargo audit`
   - `rtk cargo deny check`
2. Результат (PASS/предупреждения без blocking findings) зафиксирован в notes pack.

### AC5 (Cross-gate): release artifacts and audit checks are visible to v1 register

1. `docs/v1/gate-register.md` updated entry `last_verified_at`/`evidence` для `G3`/`G10`.
2. Все evidence-ссылки point to concrete локальные команды или конкретные файл-пути,
   а не общие абстрактные формулировки.

## Rollback / stop conditions

- Если один из CI/bench runs начинает стабильно флапаться без технического объяснения — pack
  переводится в `blocked` и приостанавливается до стабилизации infra.
- Если возникает невыполнимый regression в perf budget после фикса в runtime code — не
  закрывать G3 до устранения, пока не обновлены baseline/порог и не пересчитаны AC.

# Pack 02: Snapshot + Rollback Fail-Closed Invariants

**Status:** In Progress
**Start date:** 2026-04-15
**Related gates:** `G7`
**Blocking:** Pack 02 закрывает gate `G7`, у которого `blocking_v1 = true`.

## Purpose

Pack 02 делает верифицированной и предсказуемой ключевую часть recovery-пути v0.2.0: snapshot/rollback в Aegis должен быть fail-closed в сценариях конфиг/ID/rollback-инвариантов, а не «ложно успешным» при повреждённых или несовместимых данных.

## Scope

### Включено

1. Провайдерная инварианта для rollback по built-in провайдерам:
   - `git`
   - `docker`
   - `postgres`
   - `mysql`
   - `sqlite`
   - `supabase`
2. Инварианты registry/lookup для rollback:
   - `SnapshotRegistry::for_rollback()` и `SnapshotRegistryConfig::for_rollback_from_config()` должны приводить к полному rollback-контексту по контракту.
   - `SnapshotRegistry::rollback(...)` должен явно возвращать ошибку при неизвестном плагине.
3. Путь CLI `aegis rollback` должен fail-closed на:
   - malformed project config,
   - отсутствующий/невалидный snapshot id,
   - неконсистентный snapshot id для выбранного провайдера.
4. Расширение regression-cases для edge-failures rollback/snapshot:
   - malformed/legacy ID transitions,
   - отсутствующие артефакты (dump/file/image),
   - конфликтные rollback-сценарии (например, git stash conflict),
   - частичная деградация среды.
5. Документирование и доказательства для `G7` в `docs/v1/gate-register.md` и mini-spec.

### Исключено

- Новые snapshot providers (cloud families и т.д.).
- Policy redesign, новые snapshot patterns, или расширение attack surface вне already-specified providers.
- Новые features для конфигурации CLI и UI, не относящиеся к надежности rollback-path.

## Acceptance criteria

Pack считается завершённым только если выполнены все условия.

### AC1 (G7): rollback registry is complete and explicit

1. Модульные тесты в `src/snapshot/mod.rs` покрывают, что `SnapshotRegistry::for_rollback()` содержит полный список встроенных провайдеров.
2. `SnapshotRegistryConfig::for_rollback_from_config()` покрыт тестом, что:
   - `snapshot_policy = Full`,
   - все `auto_snapshot_*` флаги = `true`,
   - DB и sqlite/supabase параметры сохраняются из исходной конфигурации.

### AC2 (G7): rollback lookup and ID parsing are fail-closed

1. Для `aegis rollback <id>`:
   - отсутствующий id → non-zero exit и диагностическое сообщение с recovery-hint (`aegis audit --format json` / `aegis audit --last N`);
   - malformed project config → error до попытки поиска snapshot и без generic-wrapper текста.
2. Для rollback-lookup по `snapshot_id`, который формально неверный (не тот формат):
   - операция возвращает ошибку и не делает попытки silent fallback;
   - stderr должен содержать конкретное объяснение причины и имя провайдера.

### AC3 (G7): provider rollback parsing/validation is not bypassable

1. Для каждого из built-in провайдеров должен существовать regression test (unit или integration):
   - malformed snapshot id → ошибка (`malformed snapshot_id` или эквивалентный контрактный тип ошибки);
   - отсутствующий целевой артефакт (`dump`, `database`, `snapshot file`) → возвращается явная ошибка;
   - попытка rollback с некорректным target-источником не приводит к success/не-пустому «тихому» success.
2. Для провайдеров с конфликтным состоянием rollback (например, git) должен сохраняться существующий режим сохранения исходного состояния для ручного восстановления.

### AC4 (G7): command-path fails are observable and reproducible

1. На каждый важный negative rollback кейс из AC2/AC3 в pack-е есть связанный тест и артефакт/трасса, позволяющие воспроизвести.
2. Любая ошибка rollback из CLI/провайдера должна быть отражена в тестовом `stderr` и не может быть скрыта в generic-инфо (т.е. fail-closed с диагностикой).

### AC5 (Evidence): traceability from risk cases to fixes

1. Для каждого найденного regression bug/недоуловленного кейса из Pack 02:
   - добавляется failing case (или fixture);
   - после фикса — green test.
2. В `G7` notes pack включён список связок `issue → test/fixture → expected invariant`.

## Evidence requirements

Для подтверждения Pack 02 в gate register и mini-spec должны быть доступны:

- `src/snapshot/*` tests (unit): malformed/missing/rollback conflict cases для каждого провайдера в требуемом матриксе.
- Integration tests в `tests/snapshot_integration.rs` и/или `tests/full_pipeline.rs` для end-to-end rollback CLI path.
- Отчёт CLI-кейсов (`cargo test` с конкретными фильтрами) по каждому обязательному кейсу.
- Обновлённый `docs/v1/gate-register.md` (entry G7 и notes с доказательствами).

Минимальный proof bundle:

- `rtk cargo test --test snapshot_integration` (интеграционные сценарии rollback)
- `rtk cargo test --test full_pipeline rollback_` (CLI rollback сценарии)
- `rtk cargo test snapshot` (unit-уровень для provider rollback/rollback id invariants, фильтрация по имени кейсов при необходимости)
- `docs/v1/specs/v1.0-pack-02-snapshot-failclosed.md` с заполненным чек-листом

## Dependencies

- `G7` depends on `G2` (snapshot/rollback tests build on parser/scanner-hardening output from Pack 01, especially for command path resilience and edge behavior).

## Implementation notes

1. Не расширять список provider-ов в этом pack-е; покрываем только существующий built-in набор.
2. Rollback CLI-флоу не должен «маскировать» внутренние ошибки в success path: при неверном id/config/target failure должен явно завершаться как error.
3. Предпочитать table-driven/матрицу provider-failures для уменьшения дублирования тест-логики и повторного применения в будущем.

## Rollback / stop conditions

- Если появляется новый high/critical regression в rollback flow (silent success, partial recovery loss, unhandled malformed id), status Pack -> `blocked`.
- До устранения — не закрывать `G7` и не переходить к Pack 03.
- При повторяющихся flakey-тестах в snapshot-интеграциях — заморозить execution, документировать infra risk и создать отдельный action item.

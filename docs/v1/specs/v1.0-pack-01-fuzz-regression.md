# Pack 01: Parser+Scanner Fuzz & Shell Edge Regression

**Status:** Planned
**Start date:** 2026-04-15
**Related gates:** `G1`, `G2`
**Blocking:** Pack 01 закрывает gate `G1` и `G2`, оба имеют `blocking_v1 = true`.

## Purpose

Pack 01 закрывает первый критичный блок v0.2.0: сделать parser/scanner fuzzing полноценным и расширить regression suite для сложных shell-кейсов, с обязательной triage-моделью по severity.

## Scope

### Включено

1. Scanner fuzz target (дополнение/создание `fuzz/fuzz_targets/scanner.rs`) и инфраструктура запуска.
2. Интеграция fuzzing в CI/релиз-подготовку по policy для G1.
3. Добавление regression-тестов для shell edge cases в соответствующих модулях/интеграционных тестах:
   - heredoc / nested heredoc
   - inline-скрипты и кодирование спецсимволов
   - пайпы и цепочки команд
   - кавычки и escapes
   - subshell и grouping
   - многокомандные конструкции (`&&`, `||`, `;`, `{}`, `()`)
   - multiline и смешанные quoting
4. Привязка fuzz findings к regression-корпусам/регression test updates.

### Исключено

- Новые policy-поведения/риск-модель.
- Новые snapshot providers.
- Dashboards / web UI / удалённые sinks.
- Новые patterns, если для них не добавлена сопоставимая fuzz/regression coverage.

## Acceptance criteria

Pack считается готовым, только если выполнены все условия ниже.

### AC1 (G1): Fuzzing coverage exists and is reproducible

1. Scanner fuzz target существует в `fuzz_targets` и запускается локально и в CI.
2. Сборка fuzz harness-ов воспроизводится на поддерживаемой toolchain/CI образе.
3. Имеется явно указанная команда/воркфлоу для запуска (CI + локально).

### AC2 (G1): Finding triage blocks completion

1. Все fuzz findings triaged по severity: `critical`, `high`, `medium`, `low`, `informational`.
2. **Требование:** любые **неразрешённые high или critical** findings из `parser`/`scanner` fuzz автоматически переводят pack в `blocked`, даже если артефакты уже собраны.
3. Для каждого high/critical finding записывается:
   - репро (короткий)
   - ссылка на тест/issue/задачу исправления
   - статус исправления (`open` или `fixed`)
4. Pack получает статус **Passed** только после закрытия всех open high/critical и успешного перезапуска fuzz-гейта после фикса.

### AC3 (G2): Regression suite expanded with edge cases

1. Добавлены/обновлены regression-кейсы по всем перечисленным shell edge категориям.
2. Для каждого кейса есть ожидаемый риск и ожидаемый decision behavior.
3. Edge-кейсы закрывают исторические bypass/ошибки и подтверждены тестами в CI.

### AC4 (Gate evidence): Traceability from fuzz -> regression

1. Каждому подтверждённому crash/panic/false-positive из fuzz-цепочки соответствует regression test или новый fixture.
2. В notes pack указан список таких связок.

### AC5 (Performance guardrail)

1. Перед началом Pack 01 фиксируется baseline для safe-path hot-path (assess() на safe командах).
2. После изменений scanner/parser выполняется `rtk cargo bench --bench scanner_bench` с baseline-check, и новый результат для safe-path должен соблюсти целевой порог (`<= 2ms`).
3. Отчёт из benchmark-проверки приложен в evidence bundle pack-а.

## Evidence requirements

Для подтверждения Pack 01 в gate register и мини-спеке должны быть доступны:

- `fuzz` артефакт (log/run id/commit)
- CI job/step, где fuzz/gate выполняется и фиксирует результат
- список triage outcomes по severity
- перечень новых regression тестов и их pass rate
- чек-лист удаления/фиксации найденных high/critical findings

Минимальный proof bundle:

- `fuzz` command output + `cargo +nightly fuzz run scanner`
- `cargo test` subset, включающий новые edge-case regression
- `rtk cargo bench --bench scanner_bench`
- обновлённый `docs/v1/gate-register.md` (G1/G2 entries)

## Dependencies

- `G1` depends on successful parser/scanner fuzz infra.
- `G2` depends on `G1` и исправления parser/scanner hardening из того же pack-а.

## Implementation notes

1. fuzz input — это UTF-8 строки через публичный API парсера; raw-byte fuzzing вне scope этого pack-а.
2. Для стабильности fuzz harness’ов использовать минимально инвазивный путь: fuzz public parser/scanner API в том виде, в котором они используются runtime.
3. На каждый найденный finding:
   - добавить regression case в `tests/*` или fuzz corpus;
   - если воспроизводится локально стабильно — зафиксировать и закрыть перед следующей фичевой задачей в пакете.

## Rollback / stop conditions

- Если обнаружен один или более unresolved high/critical finding:
  - статус Pack → `blocked`;
  - последующие non-blocking задачи откладываются до устранения.
- При повторяющихся нестабильных fuzz crash на нестабильной окружении — зафиксировать как infra risk и временно уменьшить corpus scope только с одобрения, но без изменения security claims.
